## 1. 프로젝트 개요

### 1.1 배경 및 목적

현재 대학알리미(academyinfo.go.kr)에서 공시 데이터를 활용하려면 항목별로 파일을 개별 다운로드한 뒤, 셀 병합이 포함된 헤더를 수동으로 정리하고, 캠퍼스별로 분리 공시된 데이터를 기준 파일과 VLOOKUP으로 합산하는 과정을 매년 반복해야 한다. 이 툴은 이 반복 작업을 최소화하고, 정제된 데이터를 여러 구성원이 웹에서 바로 조회·분석할 수 있도록 하는 것이 목적이다.

### 1.2 핵심 해결 과제

- 대학알리미 엑셀 파일의 병합 헤더 자동 정규화
- 캠퍼스별 분리 공시 데이터를 기준대학 단위로 자동 합산
- 항목별 합산 방식(합계 vs 가중평균 재계산)의 체계적 관리
- 정제된 데이터를 여러 사람이 웹에서 필터·비교·시각화
- 연도별 데이터 누적 관리 및 헤더 변경 대응

### 1.3 사용자 및 권한

| 구분 | 역할 | 접근 범위 |
|------|------|-----------|
| 관리자 (1인) | 파일 업로드, 검증, 기준파일 수정, SQL 검증 | 관리자 도구 (로컬 전용) |
| 일반 사용자 (다수) | 분석 툴 조회 및 시각화 | 분석 페이지 (GitHub Pages 공개) |

> **참고** 관리자 도구는 로컬 환경에서만 실행되며 외부에 배포하지 않는다. 분석 페이지는 GitHub Pages로 배포하며, 링크를 아는 구성원만 공유받아 사용한다.

## 2. 전체 시스템 구조

### 2.1 데이터 소스 결정

대학알리미(academyinfo.go.kr/main/main0830)의 '공시데이터 추이' 페이지에서 217개 파일을 일괄 다운로드하는 방안을 검토했으나, 해당 파일의 컬럼 구조가 일반 공시자료와 상이하고 연도별 일관성이 낮아 매년 별도 정제 로직이 필요한 것으로 확인됐다. 따라서 대학알리미 공시자료(항목별 엑셀 파일)를 연 1회 수동 다운로드하여 사용하는 방식을 채택한다.

> **참고** 공식 Open API(openapi.academyinfo.go.kr)는 현재 서비스 중단 상태(JEUS 6.0 기본 화면 노출)로 활용 불가하다.

### 2.2 데이터 흐름

| 영역 | 구성 요소 | 설명 |
|------|-----------|------|
| 수집·정제 | Python GUI (normalize_gui.py) | 파일 업로드, 헤더 매핑, 캠퍼스 합산, JSON 생성 |
| 저장 | 항목별 JSON + 기준파일 JSON | GitHub 저장소에 버전 관리 |
| 분석·시각화 | GitHub Pages HTML | 순위, 비교, 추이, 쿼리 등 분석 기능 제공 |

### 2.3 저장소 파일 구조

```
/
├── CLAUDE.md                      # Claude Code 프로젝트 지침
├── index.html                     # 분석 페이지 (GitHub Pages)
├── normalize_gui.py               # 정제 도구 (Python/tkinter, 로컬 전용)
├── download_academyinfo.py        # 다운로드 자동화 (Playwright, 선택적)
├── field_mapping.json             # 필드 매핑 관리 (자동 생성/갱신)
├── data/
│   ├── 기준대학.json              # 캠퍼스 합산 기준 + 대학 속성
│   ├── 취업률.json                # 항목별 누적 데이터
│   ├── 재적학생수.json
│   ├── 중도탈락학생수.json
│   └── ...                        # 항목 추가 시 자동 생성
└── docs/
    ├── spec.md                    # 이 문서
    └── ...
```

> **성능** JSON은 항목별로 분리 저장한다. 사용자가 특정 항목 선택 시에만 해당 파일을 로드하여 브라우저 성능을 유지한다. 학과별 전국 데이터 기준 파일당 약 1~2MB 예상.

### 2.4 JSON 데이터 구조

```json

    "졸업자수": 120,

  },
  {
    "기준연도": 2024,

    "학과명": "컴퓨터공학부",
```

## 3. 관리자 기능 (로컬 전용)

> 관리자 도구는 인터넷 배포 없이 로컬에서 Python 스크립트를 직접 실행하여 사용한다.

### 3.1 정제 도구: normalize_gui.py

#### 3.1.1 GUI 구성

Python 기본 내장 tkinter 기반 GUI 애플리케이션. 윈도우 환경에서 `python normalize_gui.py`로 실행한다.

**의존 패키지**
```
pip install pandas openpyxl
```

**화면 구성 요소**

| 요소 | 설명 |
|------|------|
| 입력 폴더 선택 | xlsx 파일이 있는 폴더 선택. 선택 시 출력 폴더 자동 제안 |
| 출력 폴더 선택 | CSV/Excel 저장 위치 |
| 처리 옵션 | CSV 저장 / Excel 저장 / 항목별 JSON 누적 독립 선택 |
| 정제 시작 버튼 | 실행 중 비활성화, 완료 후 재활성화 |
| 진행바 | 전체 파일 수 기준 실시간 진행 표시 |
| 처리 로그 | 파일별 결과 색상 구분 (✅ 성공 / ❌ 오류 / ⏭ 스킵) |
| 폴더 열기 버튼 | 출력 폴더 / JSON 폴더 바로 열기 |

#### 3.1.2 병합 헤더 자동 정리

- 병합된 셀 감지 후 자동 해제
- 다행(multi-row) 헤더를 컬럼별로 `_`로 연결하여 1행으로 flatten
  - 예: `학생현황 > 학부 > 재학생` → `학생현황_학부_재학생`
- 좌상단 값을 병합 범위 전체에 채우기 (가로/세로 병합 모두)
- **데이터 시작 행 자동 감지**: 병합 해제 전, A열에서 연도(4자리 숫자)가 처음 등장하는 행을 기준으로 헤더/데이터 경계 판단
  - 세로 병합으로 A열이 비어있는 경우에도 병합 맵을 통해 좌상단 값 탐색
- **제목행 자동 제외**: 행 전체가 같은 값으로 채워진 경우(고유값 1개) 헤더에서 제외
- **시트 선택 우선순위**: Sheet1 → 이름에 'raw' 포함 → 'empty' 포함 → 첫 번째 시트

#### 3.1.3 처리 옵션

| 옵션 | 기본값 | 설명 |
|------|--------|------|
| CSV 저장 | ON | 파일별 정제 결과를 CSV로 저장. 검증 및 중간 결과 확인용 |
| Excel 저장 | OFF | 파일별 정제 결과를 xlsx로 저장. 비개발자 검토용 |
| 항목별 JSON 누적 | ON | `data/` 폴더의 항목별 JSON에 연도 데이터 append. 분석툴에서 직접 로드 |

> 같은 항목·같은 연도 데이터가 이미 JSON에 있으면 덮어쓴다. 실수로 중복 실행해도 안전하다.

#### 3.1.4 정제 파이프라인 상세

| 단계 | 처리 내용 | 개입 |
|------|-----------|------|
| 1. 시트 선택 | Sheet1 → raw 포함 → empty 포함 → 첫 번째 시트 | 자동 |
| 2. 데이터 시작 행 감지 | 병합 해제 전 A열에서 연도(4자리) 첫 등장 행 탐색 | 자동 |
| 3. 병합 해제 + fill | 모든 병합 셀 해제 후 좌상단 값으로 전체 채움 | 자동 |
| 4. 헤더 flatten | 헤더 행들을 컬럼별 `_` 연결로 1행 축약 | 자동 |
| 5. 필드 매핑 적용 | field_mapping.json 역방향 조회로 alias → 표준 필드명 | 자동 |
| 6. 새 필드 확인 | 매핑에 없는 필드 발견 시 팝업 표시 | **사람 개입** |
| 7. 항목 키 추출 | 파일명 앞 연도(`YYYY년_`) 제거 후 나머지를 항목 키로 사용 | 자동 |
| 8. CSV/Excel 저장 | 옵션 ON 시 출력 폴더에 파일별 저장 | 선택 |
| 9. JSON 누적 | 옵션 ON 시 `data/{항목키}.json`에 연도 데이터 append | 선택 |

### 3.2 필드 매핑 관리 (field_mapping.json)

#### 3.2.1 팝업 UI

새 필드 또는 이름이 바뀐 필드 발견 시 자동으로 팝업이 표시된다.

| 처리 방식 | 설명 | 예시 |
|-----------|------|------|
| 기존 필드에 매핑 | 이름이 바뀐 경우 기존 표준 필드명으로 연결 | `취업자수_합계` → `취업자수_계` |
| 새 필드로 추가 | 완전히 새로운 항목 — 표준 필드로 신규 등록 | `해외취업자수` 신설 |
| 이 파일만 무시 | 임시 컬럼 등 누적 불필요한 경우 해당 파일에서만 제외 | 비고, 임시메모 등 |

- 확인 시 결정 내용이 `field_mapping.json`에 자동 저장 → 다음 연도엔 자동 처리
- 취소 시 해당 파일 스킵

> 헤더 변경이 없는 일반적인 연도에는 업로드 → 자동 처리 → JSON 누적까지 팝업 없이 완료된다.

#### 3.2.2 field_mapping.json 구조

```json
{
  "취업자수": [
    "취업자수_계",
    "취업자수(계)",
    "취업자수_합계"
  ],
  "재학생수": [
    "재학생(A)",
    "재학생수_계"
  ],
  "해외취업자수": []
}
```

표준 필드명(키) → 지금까지 쓰인 별칭(alias) 목록 구조. 새 별칭이 추가될 때마다 자동 갱신된다.

#### 3.2.3 항목 키 추출 규칙

- 파일명 앞의 연도 패턴(`YYYY년__`) 제거 후 나머지를 항목 키로 사용
- 예: `2024년__대학_6-나-(1)_전임교원_확보율_학과별자료.xlsx` → `대학_6-나-(1)_전임교원_확보율_학과별자료`
- 같은 항목의 연도별 파일은 파일명 앞 연도만 다르고 나머지가 동일해야 같은 JSON에 누적됨
- 파일명이 다를 경우 관리자가 직접 파일명을 맞춰서 처리

### 3.3 캠퍼스 합산

#### 3.3.1 합산 기준

- `기준대학.json`의 `대학명 → 기준대학명` 매핑을 참조하여 캠퍼스를 그룹화
- 예: `연세대학교(서울)`, `연세대학교(미래)` → `연세대학교`로 합산

#### 3.3.2 항목별 합산 방식 (merge_rules.json)

JSON에는 원본 자수(raw data)만 저장한다. 비율 지표(취업률, 충원율 등)는 저장하지 않고 분석 페이지에서 즉석 계산한다.

| 합산 방식 | 적용 항목 예시 | 비고 |
|-----------|---------------|------|
| `sum` | 재적학생수, 취업자수, 졸업자수, 중도탈락학생수 등 자수 | 캠퍼스별 값 합산 |
| `skip` | 취업률, 충원율 등 비율 지표 | JSON에 저장하지 않음 |
| `master` | 설립구분, 대학구분, 지역 등 속성 | 기준파일 값 사용 |

```json
// merge_rules.json 예시
{
  "재적학생수":   { "method": "sum" },
  "취업자수_계":  { "method": "sum" },
  "졸업자수":     { "method": "sum" },
  "취업률":       { "method": "skip" },
  "설립구분":     { "method": "master" }
}
```

> **미결** 20개 항목 각각의 합산 방식 정의 필요 (9. 미결 사항 참조)

#### 3.3.3 비율 지표 산식 정의 (calc_rules.json)

비율 지표의 산식은 `calc_rules.json`에 정의한다. 분석 페이지(JS)가 이 파일을 읽어 즉석 계산한다. 산식이 변경되더라도 이 파일만 수정하면 코드를 건드리지 않아도 된다.

```json
// calc_rules.json 구조 예시
{
  "취업률": {
    "label": "취업률(%)",
    "numerator": ["취업자수_필드1", "취업자수_필드2"],
    "denominator_base": "졸업자수",
    "denominator_exclude": ["제외자_필드1", "제외자_필드2"],
    "multiply": 100
  }
}
```

> **미결** 항목별 정확한 산식(분자/분모 필드명) 정의 필요 (9. 미결 사항 참조)

### 3.4 검증

#### 3.4.1 자동 검증 항목

- 필수 컬럼 누락 여부
- 숫자 필드에 문자값 혼입 여부
- 대학명이 기준파일에 없는 경우
- 전년도 대비 총 행 수가 10% 이상 변동된 경우 경고
- 특정 대학/학과의 데이터가 전년도에 있었다가 사라진 경우 목록 표시
- 캠퍼스 합산 후 합계값이 원본 합계와 일치하는지 교차 검증

#### 3.4.2 검증 결과 표시

- 오류/경고/정상 3단계로 구분하여 로그 색상 표시
- 문제 있는 행을 테이블로 표시 (대학명, 학과명, 오류 내용)
- 오류가 있을 경우 JSON 생성 전 확인 팝업 표시

### 3.5 SQL 검증 툴

- sql.js 라이브러리를 활용하여 브라우저에서 SQLite 쿼리 실행
- 생성된 JSON 파일들을 인메모리 SQLite DB로 로드
- SQL 입력창에서 직접 쿼리 작성 및 실행
- 쿼리 결과를 테이블로 출력, CSV 다운로드 지원
- 자주 쓰는 검증 쿼리 저장 및 재사용

> 이 기능은 관리자가 데이터 품질을 직접 검증하기 위한 도구이며, 일반 사용자 분석 페이지에는 노출되지 않는다.

## 5. 분석 페이지 (GitHub Pages)

여러 구성원이 링크로 접근하여 사용하는 웹 분석 툴이다. 서버 없이 정적 HTML로 동작하며, 항목 선택 시 해당 JSON 파일만 동적으로 로드한다.

### 5.1 공통 필터 (글로벌)

모든 분석 탭에 공통 적용되는 상단 필터.

| 필터 항목 | 유형 | 선택지 |
|-----------|------|--------|
| 공시 항목 | 드롭다운 (단일) | 취업률 / 재적학생수 / 중도탈락학생수 / ... |
| 연도 | 드롭다운 (단일 또는 범위) | 데이터 있는 연도만 표시 |
| 대학구분 | 드롭다운 (다중) | 4년제 / 전문대 / 대학원대 |
| 설립구분 | 드롭다운 (다중) | 국공립 / 사립 |
| 지역 | 드롭다운 (다중) | 서울 / 경기 / 부산 / ... |
| 수도권 여부 | 토글 | 전체 / 수도권 / 비수도권 |

### 5.2 순위 보기

- 공통 필터 적용 후 전체 대학을 선택 항목 기준으로 정렬하여 순위 테이블 표시
- 우리 대학은 별도 색상으로 하이라이트
- 현재 우리 대학의 순위와 상위 백분율(%) 상단에 크게 표시
  - 예: `전국 4년제 사립대 기준 취업률 상위 12.3% (23위 / 187개교)`

**테이블 컬럼**

| 컬럼 | 설명 |
|------|------|
| 순위 | 필터 적용 후 현재 정렬 기준 순위 |
| 대학명 | 기준대학명 (클릭 시 상세 팝업) |
| 지역 / 설립구분 | 기본 속성 정보 |
| 값 | 선택한 항목의 수치 (자수 및 비율 모두 표시) |
| 전년 대비 | 전년도 대비 증감 (▲▼ 표시) |
| 상위 % | 현재 필터 그룹 내 상위 백분율 |

- 컬럼 클릭으로 오름/내림차순 전환
- 페이지당 50개 표시, 페이지네이션 제공
- CSV 다운로드 버튼 제공

### 5.3 그룹 평균 비교

- 우리 대학의 수치를 선택한 경쟁 그룹의 평균과 나란히 비교
- 경쟁 그룹은 공통 필터로 정의 (예: 수도권 4년제 사립대)
- 막대 차트: 우리 대학(강조색) vs 그룹 평균(회색)
- 평균 대비 우리 대학의 차이(+/-) 수치 레이블 표시

### 5.4 연도별 추이 비교

- 선택한 대학의 항목을 연도별 꺾은선 그래프로 시각화
- 비교 대학 추가 선택 가능 (최대 5개), 우리 대학은 굵은 선으로 강조
- 대학명 검색 후 추가, 칩(chip) UI로 개별 제거 가능
- 공통 필터 그룹의 연도별 평균을 점선으로 오버레이 (토글)

### 5.5 대학 간 직접 비교

2~5개 대학을 선택하여 항목별로 나란히 비교한다.

**항목 하나, 여러 대학 비교 → 가로 막대 차트**
- 선택한 항목 기준으로 대학별 수치를 가로 막대로 표시
- 우리 대학 막대 강조색, 나머지 회색
- 막대 끝에 수치 레이블 표시

**여러 항목, 여러 대학 비교 → 히트맵 테이블**
- 행: 대학, 열: 항목 (여러 항목 동시 선택 가능)
- 셀 색상: 값의 상대적 크기를 색상 농도로 표현 (높을수록 진함)
- 우리 대학 행 강조 표시
- 숫자 직접 비교가 가능하도록 수치 병기

> 레이더 차트는 단위가 다른 항목들을 하나의 축에 정규화해야 하는 문제가 있어 사용하지 않는다.

### 5.6 학과별 드릴다운

- 대학 선택 후 해당 대학의 학과별 데이터를 테이블로 표시
- 학과명 검색 및 계열별 필터 지원
- 학과별 순위 (해당 대학 내 순위 및 전국 동일 계열 학과 중 순위) 표시
- 학과 선택 시 연도별 추이 팝업

### 6.1 디자인 시스템: KRDS 준수

본 툴의 UI는 행정안전부 범정부 디자인 시스템인 **KRDS**(KoRea Design System, krds.go.kr)를 기준으로 설계한다.

차트(Chart.js/ECharts)처럼 KRDS에 없는 컴포넌트는 KRDS 색상·폰트 토큰을 적용하여 일관성을 유지한다.

### 6.2 반응형 설계

분석 페이지는 PC, 태블릿, 모바일 모두에서 정상 사용 가능하도록 설계한다. 관리자 도구(Python GUI)는 PC 전용으로 설계한다.

**브레이크포인트**

| 구분 | 화면 너비 | 주요 대응 |

## 7. 기술 스택 및 배포

### 7.1 기술 스택

| 영역 | 기술 | 선택 이유 |
|------|------|-----------|
| 데이터 정제 | Python (pandas, openpyxl, tkinter) | 엑셀 병합 처리, GUI 내장 라이브러리 |
| 다운로드 자동화 | Playwright (Python) | Canvas UI 브라우저 자동화 |
| 데이터 저장 | JSON (항목별 분리) | 서버 없이 브라우저에서 직접 로드 |
| 분석 페이지 | HTML + Vanilla JS | 서버 불필요, 브라우저만으로 동작 |
| 차트 | Chart.js 또는 ECharts | 무료, CDN, 한국어 지원 |
| SQL 검증 | sql.js (SQLite WebAssembly) | 브라우저에서 서버 없이 SQL 실행 |
| 배포 | GitHub Pages | 무료, 링크 공유로 접근 |

### 7.2 JSON 누적 규칙

- 기존 JSON에 없는 연도 → append
- 기존 JSON에 같은 연도가 있으면 → 해당 연도 전체 덮어쓰기 (중복 실행 안전)
- 새 연도에 추가된 필드는 이전 연도 레코드에서 `null`로 유지

### 7.3 다운로드 자동화 스크립트 (download_academyinfo.py)

대학알리미 공시데이터 추이 페이지는 Canvas 기반 UI로 구성되어 일반 HTTP 요청으로는 파일 다운로드가 불가하다. Playwright 브라우저 자동화 스크립트를 제공한다.

**기술적 특이사항**

- 다운로드 버튼이 Canvas 위에 그려진 UI → CSS 셀렉터 사용 불가
- 각 행의 클립 아이콘을 픽셀 좌표로 클릭 (`x=1056, y=65 + 행번호 × 42`)
- 클릭 시 팝업 새 창 열림 → 사유 입력(`textarea#rsnDetail`) → 다운로드 버튼(`#rsnSave`) 클릭
- 다운로드는 팝업 닫힌 후 `page.goto()` 호출 시 트리거되는 구조

**설정값**

| 설정 | 기본값 | 설명 |
|------|--------|------|
| TOTAL | 217 | 전체 파일 수 |
| CANVAS_X | 1056 | 다운로드 버튼 열 x좌표 (고정) |
| FIRST_ROW_Y | 65 | 1번째 행 y좌표 |
| ROW_HEIGHT | 42 | 행 높이 px (codegen으로 확인) |
| TEST_MODE | True | True: 처음 3개만 / False: 전체 |
| START_FROM | 1 | 중간 실패 시 재시작 행 번호 |
| DELAY | 1.0 | 파일 간 딜레이 (초) |

> 연간 공시자료는 수동 다운로드로도 충분하므로 이 스크립트는 선택적으로 활용한다.

### 7.4 배포 방식

- GitHub 저장소에 `data/` 폴더와 `index.html` 저장
- GitHub Pages로 정적 호스팅 (무료)
- Python 스크립트로 JSON 업데이트 후 `git push` 하면 분석 페이지 자동 갱신
- 분석 페이지는 링크를 아는 구성원만 공유받아 사용 (별도 인증 없음)

### 7.5 연간 운영 흐름

| 시점 | 작업 | 소요 예상 시간 |
|------|------|---------------|
| 매년 공시 갱신 후 | 대학알리미에서 항목별 파일 (~20개) 다운로드 | 20~30분 |
| | `normalize_gui.py` 실행 → 입력 폴더 선택 → 정제 시작 | 5분 |
| | 새 필드 발견 시 팝업에서 매핑 확인 (없으면 자동 통과) | 0~10분 |
| | CSV/Excel로 결과 검증, SQL 검증 툴 활용 | 10~30분 |
| | JSON 누적 확인 → `git push` → 분석 페이지 갱신 | 5분 |
